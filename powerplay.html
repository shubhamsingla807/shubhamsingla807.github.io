Proccess

The process is simple, we first identify what‚Äôs the problem, then layout a Epic, map things with the definition of our Implementation POD, work on solving the problem starting with user research, putting down user journey‚Äôs, their goals & needs and finally the multiple solutions to test with the users. If I had to write it in the process, then it would go like -

Now as a part of Implementation POD, we have our definition with us and we need to move those numbers. So, we need to build the plan on how to improve these numbers. Our POD focuses on to make sure users make an investment in the product so that they are ready to get retained & became an active user in the product.

Okay let‚Äôs first understand a basic flow of user journey, so that you could have a better idea -

  • User signs up on the application.
  • User was prompted to create a new site. (As a user from construction industry, they used to create sites. Here site denotes the place where construction work is going on).
  • User created a site.
  • User was asked to create different tasks. (Here tasks denotes the site work that construction industry workers do for ex - Bricking, Lintel, Foundation etc.)
  • User creates a task and do a lot of other stuff as well.

After this there are many more things a user does but here, I didn‚Äôt go into the much detailed journey as my POD comes under this journey only, when user created a site our main motive is to make sure user creates more tasks & realize the value of our application. Now to make sure user do the intended action, we devised a strategy.

New strategy

Along with providing value to our user we need to make sure user also makes an investment in our product . To do so, we came up with two major epics one is ‚ÄúPre-built templates‚Äù & other is ‚ÄúUpload Plan‚Äù. Now by names it would be hard to identify what exactly are these, I‚Äôll try to explain them below -

Pre-built templates - Currently user had to manually create every task and generally in construction industry there are a lot of tasks, so our assumption was if we help users built tasks easily with our templates then we can motivate them to build more & help them to stay active on the platform.

Upload plan - We found that users already had some site plans & tasks created with them either in the form of Excel or PDF which they used to communicate with other members of their team. If we were able to get them their plans uploaded on our application, then the burden of creating tasks from scratch would be over and they could easily use our application.

Currently, I‚Äôm working on Pre-built templates feature, so further would be explaining the same.

MVP Building

Now we have the problem, to solve the same we need to build hypothesis based upon user journey & insights. Before simply building the complete design, we also validated whether showing templates is a good idea or not? We build a small MVP to test the same and the numbers are good, then we proceeded with multiple nudge points and use cases.

In the above experiment, the task accuracy is not that good, but we got to know user actually find them useful and to make it better, we decide to create different nudge points where it provides more value to user, also conducted user calls to get better clarity on the type of tasks. Also, we‚Äôre showing templates to these users once they at least added a single task.

In this, ‚ÄúTest‚Äù is the data of the experiment and ‚ÄúControl‚Äù is the data of general flow. Clearly activation of the experiment is higher than the general flow over the same period of time.

Building multiple hypothesis and experiments

Now after this there are certain places where we can provide different value to the user based upon multiple nudge points. To do the same, we need to identify those values and needs to show it to the user. The point is wherever we saw the more usage of templates, we could say that this is the place where user finds the more value and could be used further for 100% rollout.

1. Site Creation (Prioritized One - 1st Flow)

For the very first time, we decided to go with showing templates to the users at the time of 1st site creation, reason being is this is already tested and had given good numbers, apart from that this is the place where there are more eyes and help us adoption in templates.

Hypothesis - ‚ÄúAs a user, I know what tasks to create, but there are a lot of tasks to create manually, if I‚Äôm able to choose from predefined tasks list, then creating new tasks would be easy and effortless for me.‚Äù

Certain changes that we did in this from the past exp is improvement in design + improvement in suggested tasks. Also, earlier suggestions are shown in a way, that it‚Äôs uncomfortable for the user and once canceled these couldn‚Äôt be viewed again. Now suggestions are shown in a way in which user can hide and open them as needed. Moreover, these suggestions would disappear once a user adds a minimum of 15 tasks or click on ‚ÄúAdd all tasks‚Äù.

Why 2 flows?

We have created two flows, to verify & understand at what placement or in which scenario the user tends to use templates more. The better we got implementation %, the better are the chances users would stick to the platform.

In the 1st scenario, we are showing suggestions to the user, allowing him/her to add tasks at any moment, while creating tasks. In the 2nd scenario, we‚Äôre asking users to pre-select tasks early and then showing selected tasks directly in the tasks list. (Here selecting at least a single task is mandatory to proceed for the user).

The 2nd version of the same was not prioritized for now, reason being is in this we are asking users to select at least one of the task initially to proceed, which is a risk for us, as if user doesn‚Äôt find a particular task then chances are high user won‚Äôt proceed further. Thus, we rejected this flow for the time being.

2. At the time of 2nd Site Creation

Now instead of showing templates on 1st site, our assumption is user had realised the value of PP and now starts creating 2nd site, but now it seems like a pain for user, as adding repetitive tasks is not easy for the user, so allowing users to onbaord tasks from past site well along with suggestions.‚Ä®

Hypothesis - ‚ÄúAs a user, I want to create another site, but creating tasks from beginning is difficult, if I can edit/onboard tasks from my previous site, then I can create new site with easily and can explore more.‚Äù

3. When user had created his 1st task

Again here our assumption is user had created one task and realised creating more tasks is hard, so if at the time when user had created his task, he would able to better realise the value of templates. ‚Ä®

Hypothesis - ‚ÄúAs a user, I created one task, but if I can create multiple tasks, then my chances of creating more tasks would increase and help me completing my project.‚Äù

Apart from these there are certain different nudge points are there, but these are the more important ones. Now the next thing is the priortization, among all these we need to indentify one that would go live for the new users.

From all of these we prioritized the first one reason being is -

  • It was already tested and had given good results
  • More eyes are on this page (User engagement)
  • First phase where user realised the pain of creating multiple tasks

Also for this experiment one of the riskiest assumption we have taken is that ‚ÄúWhat if users didn‚Äôt find the tasks relatable according to their work?‚Äù To make sure all tasks are fine, we conducted 1-1 user calls + check data to create the new task list that resonates with the users. First we conducted user calls and for that we got the data of users from business & success team.

User calls & data analysis

The only thing that remains little difficult for me in this industry is user calls, yes I love talking with users but in construction industry it‚Äôs very hard to get insights from the users, main problem arises when users didn‚Äôt answers the call. Even I had to drop a few calls because of the language they speak. But after conducting 30+ user calls, we were able to get some insights on the type of tasks they create.

Goal - To better understand what types of tasks a user creates and how their current process looks like. What kind of tools do they use to create tasks or do things?

Here are the questions I asked, I formated the questions in Hindi, as users understand that language only -‚Ä®

  • Aap currently kis team (like office ya site) ki side se kaam krte hn or aapka role kya hai currently?
  • Aapka day to day role smjhaynge ek baar ki, kaise aap day start krte hn or kya task hota hai generally jab aap kisi project pr kaam krte hn?
  • To jaise ki aap road construction(/industrial) projects pr kaam krte hn, to usme aap project ko kaise initate krte hn, matlab sabse phli cheeg kya hoti hai fir next & next? (Deep dive here)
  • Kis type ke tasks aap create krte hn initially, any examples?
  • Project complete hone ke baad, aap kya krte hn next?

From these user calls, the main motive is to understand the type of tasks they create and how‚Äôs their journey looks like. Upon overall analysis we had created a set of tasks in multiple categories such as Residential, Commercial etc. Apart from this we also checked the tasks created by users in the application already, we segregated data and build clusters based upon similarity of the type of project. But, the tasks we were able to identify are the general one and not that deep.

Results & further iteration

This experiment works very well and we got some good numbers, there‚Äôs almost 12% increase as compared to the past numbers. Our 10+ Task creation rate almost becomes 12 times, but the task log % is not that high but still better than the past one‚Äôs.

We have also checked the retention data of that cohort and that‚Äôs also positive and had increased as compared to the platform wise, but again number is not that high. So, we go through the user recordings and did a lot of user calls to understand user behaviours and their expectations from the templates. Upon completely analysing things, these are the major insights that come out -

  • Users are not clicking on Suggestion Icon card, which is cluttering the space right now.
  • Unidentified Value of templates - Currently users think that Adding all tasks is mandatory, and are unable to find the value of templates.
  • Multiple nudges - Currently users are adding tasks and get nudges by other features and leave the application. Certain things that distracts the user - Update task, Invite site team, Get demo.

To solve all these challenges, we tried to modify a few things and test it again with the users, here are the changes -

Now, in the first design we‚Äôre trying to tell the value of templates and how it could help them, in the 2nd screen as soon as user adds a task we‚Äôre nudging user with the ‚ÄúUpdate task‚Äù coach mark, this would help the user to remain in a natural flow and helps us to increase the task logs as well.

For the 1st problem of suggestion card, we had given elevation to the card + make the icon bigger, so that user could differentiate between the task lists and the suggestion card.

After this we tried few more versions as well, to test multiple things, and finally we did our last experiment i.e. V6, in this we have categorized tasks structurally and give proper categories to the tasks, and we believe this would perform the best, but results shows us something else -

When we saw the results after 1 week, we saw our quality metric is down as compared to the ‚ÄúControl‚Äù version. Also, the base metric i.e. 10+ Task creations is also low as compared to past experiments, but better than the control. So, we did a RCA to understand why this is happening, two major reasons come into the notice -

  • Important/Most Created tasks are far below in the suggestions list, previously we have put the most created tasks on top but in this experiment we didn‚Äôt or say forgot to do that (Everyone do mistakes :p). So, now we think because of this user is unable to find the required task and found templates unhelpful.
  • Second reason is, from Mixpanel we identified the major project category that users are choosing is ‚ÄúOthers‚Äù and in that category the tasks that are shown are fundamentally wrong, as for each task, backend team created a new category that leads to a lot of horizontal scrolling. This happens due to some miscommunication, later we corrected it.

After improving the 2nd critical issue, we saw our base & quality metric comes to normal, but still the retention data looks low as comapred with the Control.

Overall our numbers look good, so we decided to launch this experiment to 100% of the users, but before that we need to check with all other PODs/team members to identify how launching might affect their numbers, upon collaborating with all team members, we got to know Activation POD launches an experiment as well that collides with our user journey, so to resolve the same, we did A/B testing where we want to indentify which user flow helps us build the best for our users.
